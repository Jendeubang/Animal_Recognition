| 选项 | 选择 | 理由 |
|------|:----:|:-----|

### 4.2 故意制造失败

#### 失败方式 1：修改测试文件使断言失败

```python
# 在 test_ci.py 中故意将断言改错
assert False, "故意制造失败"
```

**运行结果**：流水线在测试阶段中止，输出错误日志

**控制台日志片段**：
```
FAILED python/test_ci.py::test_imports - AssertionError: 故意制造失败
```

#### 失败方式 2：删除依赖包

```bash
pip uninstall fastapi -y
```

**运行结果**：安装依赖后测试时，模块导入失败

#### 修复过程

```bash
# 方案1：恢复测试文件
git checkout python/test_ci.py

# 方案2：重新安装依赖
pip install -r python/requirements.txt
```

## 五、质量门禁与通知（任务五）

### 质量门禁配置

| 门禁规则 | 配置方式 | 触发动作 |
|:---------|:--------|:---------|
| **测试失败中止** | GitHub Actions 默认 fail-fast | 后续阶段不执行 |
| **语法错误阻止** | 编译/语法检查失败即中止 | 流水线停止 |
| **依赖安装失败** | pip/mvn 返回非0退出码 | 流水线停止 |
| **构建产物归档** | `archiveArtifacts` (Jenkins) | 保存 jar 包 |

### 通知方式

| 方式 | 说明 |
|:----|:------|
| **控制台输出** | 实时查看各阶段日志 |
| **GitHub 邮件** | push/PR 结果自动通知 |
| **钉钉/飞书 Webhook** | 可通过脚本发送通知 |
| **Jenkins 邮件** | 配置 `mail` 步骤 |

### CI/CD 对项目的影响

| 维度 | 没有 CI/CD | 有 CI/CD |
|:----|:-----------|:---------|
| **代码质量** | 靠人工检查，容易遗漏 | 自动化测试+语法检查 |
| **协作效率** | 合并前手动测试 | push 自动验证 |
| **问题发现** | 上线后才发现 | 提交时即时发现 |
| **构建一致性** | 依赖个人环境 | 统一环境运行 |
| **交付速度** | 手动打包部署 | 一键构建归档 |
